Use this flow and stop as soon as the most likely failing layer is identified:

0. Define problem
- Collect exact error, affected scope, and start time
- If the user already gave reliable results for early checks, do not repeat them

1. DNS
- Commands: nslookup <hostname>, dig <hostname>
- Fail -> DNS issue
- Pass -> Network reachability

2. Network reachability
- Commands: ping <host>, traceroute <host>
- Fail -> routing/firewall issue
- Pass -> Port connectivity

3. Port connectivity
- Commands: nc -zv <host> <port>
- Fail -> SG/NACL/listener issue
- Pass -> HTTP check

4. HTTP check
- Commands: curl -vk https://<host>/health or curl -v http://<host>:<port>/health
- Timeout -> LB/network path
- 5xx -> App/logs
- 403/401 -> IAM/auth
- 200 -> Dependency/config path

5. App logs
- Command examples: journalctl -u <service> -n 200, kubectl logs <pod> --tail=200
- Error found -> fix app issue
- No error -> OS health

6. IAM/Auth
- Check IAM policies, auth gateway, token expiry, WAF/API gateway rules

7. OS health
- Commands: top, free -m, df -h
- Not OK -> fix resources
- OK -> Process check

8. Process check
- Commands: ps, systemctl status <service>, ss -tuln
- Not running -> restart service
- Running -> Cloud infra

9. Cloud infra
- Check LB target health, SGs, NACLs, ASG health

10. Dependency check
- Check DB, cache, and external APIs

11. Config drift
- Compare current config/state with last known good
